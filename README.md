
# 12 AI Skills for Product Managers

A library of reusable AI skill files for enterprise technology product managers. Each skill is a structured `.md` file that encodes the framework, best practices, output format, and guardrails for one specific PM activity.

**A prompt is a moment. A skill is an asset. And assets compound.**

---

## What this is

Most product managers use AI the way you'd use a clever intern who forgets everything between conversations. A prompt here, a rewrite there, help that vanishes the moment the chat window closes.

This library is built on a different idea: instead of typing a new prompt every time, you build a reusable skill — a structured file you reach for the same way you'd reach for a trusted framework. Every time you need a discovery synthesis, a roadmap tied to measurable outcomes, or an executive narrative, you load the skill and get consistent, high-quality output.

These files are designed for product managers working inside large or regulated enterprises, where speed without structure is a liability. Best practices are baked in. Governance can be layered on. The output is consistent across a whole team.  Be sure to spend the time to make these relevant to your work situation in order to experience the value they can bring.

---

## How to use these files

1. Open the skill file for the activity you're working on
2. Each skill will have 3 files (README, Skill File, Example Output).  Be sure to start with the README file for all of the instructions on that skill
3. Copy the files down to your own system for editing and use 
4. Save your customized version separately — keep the base file clean for reuse

> **Note:** These are generic base files. They are designed to be customized for your organization, product domain, and team. Never paste proprietary or sensitive information into a public AI tool.

## The enterprise PM workflow — why the sequence matters

Most PM frameworks organize skills into neat phases: Discover, Define, Build, Launch. Clean on paper. Not how it works inside a large organization.

This library sequences the 12 skills the way enterprise product management actually happens — where funding gates exist, where engineering needs to be a partner from day one, and where committing the organization to build something before you have leadership alignment is a liability, not a shortcut.

Here is the logic behind the sequence:

**1. Start with understanding the problem.**
Product Discovery comes first. Always. You cannot set strategy, build anything, or ask for funding until you deeply understand what problem you are actually solving.

**2. Prototype early to confirm your understanding.**
Before strategy is locked, get something tangible in front of the business. Prototyping at this stage is not about building — it is about validating that you have understood the problem correctly. It also brings engineering in as a partner from the very beginning, not a downstream recipient of specs.

**3. Set strategy once you know what you are solving and roughly how.**
Product Strategy comes after Discovery and Prototype, not before. Direction informed by real evidence is strategy. Direction set in a vacuum is a guess.

**4. Secure leadership buy-in before committing to build.**
Executive Storytelling is not a launch activity. In an enterprise, it is a gate. There is no point completing a roadmap, writing specs, or evaluating vendors if leadership has not funded and aligned on the direction. Get the runway secured first.

**5. Plan and specify with confidence.**
Roadmap Development and PRD Generation come after funding is secured. Now you are planning with authority, not speculation.  Your prototype should be evolving into the solution you're looking to enable and will inform the next several steps.

**6. Evaluate build vs. buy before you start building.**
Vendor & Platform Selection happens before Evaluation and Analytics — because what you are building, and with what tools, heavily influences every decision that follows.

**7. Build, test, measure, and scale.**
AI Evaluation & Experimentation, Product Analytics, and Product Operating Model follow in sequence — test what you are building, measure what you learn, then design the operating model that sustains it.

**8. Go to market.**
Go-To-Market Planning closes the build cycle and opens the launch.

**9. Sustain and compound.**
The AI PM Coach is the capstone — an ongoing skill that ties the entire system together and keeps it improving.

If you are building software inside a large enterprise and you want consistent, governed, high-quality AI output across your entire PM workflow, this library is for you.  Even if you're in a different industry or a startup, these skills could be modified and applicable.

## A note on customization

Each base skill file is intentionally generic. That is by design — generic files are portable, shareable, and team-ready.

When you apply a skill to your own context:
- Create a copy of the base file
- Add your relevant context, terminology, and constraints in the bracketed sections
- Version your customized file separately
- Keep the base file untouched for future updates
